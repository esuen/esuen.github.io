Astral Rift

A real-time top-down space combat game built in Unreal Engine 5 C++ using AI-assisted development.

Unreal Engine 5 C++ HLSL Python Niagara VFX Gameplay Ability System

Why I built this

In the early 2000s, Sony Online Entertainment released Cosmic Rift—a top-down spaceship combat game where your health and your ammo were the same resource. Every shot you fired made you more vulnerable. That tension—spend energy to kill, or save it to survive—produced some of the most compelling competitive gameplay I've experienced. The servers shut down years ago, and nothing replaced it.

I built Astral Rift using Claude Code as an AI development partner. Starting from zero UE5 experience, I went from an empty project to a full combat game with 8 ship types, AI opponents, physics-based black holes, and a complete map system—all in pure C++, zero Blueprints.

Game design

Energy is everything

A unified energy system where health, ammo, and abilities draw from the same pool. Fire a weapon and you lose health. Activate overdrive and you drain faster. Every decision has a cost.

8 ship archetypes

From the Warbird (precision duelist) to the Shark (no guns, six mines). Each ship has unique weapons—bouncing bombs, shrapnel explosions, EMP blasts that drain energy and suppress regen. Data-driven—all stats and abilities defined in asset files, not code.

Environmental hazards

Black holes that curve projectiles and teleport ships. Nebula clouds that create fog-of-war—hidden ships can only see nearby. Procedural asteroids as destructible cover.

Bounty system

Kill streaks increase your bounty, making you worth more points but also a bigger target. Natural rubber-banding that keeps matches competitive without artificial handicaps.

Technical highlights

Pure C++ on UE5

Zero Blueprints. Full Gameplay Ability System integration—energy as a GAS attribute, weapons and items as GameplayAbilities, damage via GameplayEffects, state queries via GameplayTags.

AI combat system

6-state utility AI with Reynolds steering behaviors, predictive aiming, bomb trajectory simulation, and enemy energy estimation. Environmentally aware—navigates around walls, avoids black holes, retreats into nebula clouds for cover, and uses the event horizon as an escape route when losing fights.

Custom HLSL shaders

Black holes rendered with Schwarzschild-inspired spacetime distortion and Keplerian particle orbits. Nebula clouds use fractal Brownian motion with Kolmogorov turbulence. Asteroid surfaces generated with fBm noise and Voronoi crater patterns.

Data-driven architecture

Ships, weapons, maps, and hazards all defined as UPrimaryDataAssets. Text grid map editor with a Python import pipeline. In-game editor with undo, JSON serialization, and chat commands.

Performance engineering

29 optimizations including projectile object pooling (zero allocations during combat), a centralized actor registry replacing per-frame world scans, tick budgeting, and material instance reuse across pooled actors.

Complete game systems

Match flow (warmup, timed rounds, results), team assignment, kill feed, HUD with minimap, ship selection menu, options with key rebinding, bounty-based scoring, weighted item spawning, and an action-scoring spectator camera that auto-follows the most interesting combat.

How it was built

I built this using Claude Code as an AI development partner. I drove all architecture, game design, and systems decisions. Claude handled implementation—writing C++, HLSL shaders, Python tooling—while I directed, tested in-engine, and iterated. The entire codebase is C++ with zero Blueprints, running on Unreal Engine 5.7.

Custom HLSL shaders, a 6-state AI system with environmental awareness, full GAS integration, ship-specific abilities, a data-driven map pipeline, and performance-optimized combat. AI-assisted development didn't lower the bar—it let me operate at a pace that would normally require a small team.

My background is a decade of backend engineering in Go and Python—distributed systems, internal platforms, CI/CD. This project proved that systems thinking transfers directly: the same principles that make a distributed backend resilient (separation of concerns, data-driven configuration, clear ownership boundaries) make a game codebase maintainable. The difference is your users notice latency in milliseconds, not seconds.
